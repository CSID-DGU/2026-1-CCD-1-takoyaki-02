# 새 게임 추가 가이드 (Adding a New Game)

본 문서는 시스템에 새로운 보드게임을 추가하기 위해 수정해야 하는 구성 요소와 작업 순서를 정의한다.
시스템 구조에 익숙하지 않은 개발자도 따라올 수 있도록, 먼저 전체 동작 원리와 핵심 개념을 설명한 뒤
구현 단계를 안내한다.

대상 독자: 본 프로젝트에 처음 합류했으며 새 게임 모듈을 추가하려는 개발자.
선행 지식: Python 기본 문법. 시스템 내부 구조에 대한 사전 지식은 필요하지 않다.

요트다이스(`yacht`)와 한밤의 늑대인간(`werewolf`)이 동일한 골격을 공유하므로,
두 게임 중 추가하려는 게임과 성격이 가까운 쪽을 복사하여 시작하는 것을 권장한다.

---

## 1. 시스템 개요

본 시스템은 사람 진행자 없이 보드게임을 진행하는 AI이다. 기존에 사람 진행자가 수행하던 역할을
네 개의 구성 요소가 분담한다.

| 진행자의 역할 | 구성 요소 | 디렉터리 |
|---|---|---|
| 테이블을 관찰하여 상황을 파악 | 비전(카메라 인식) | `vision/` |
| 규칙에 따라 진행과 판정을 결정 | 게임 규칙 엔진(FSM) | `games/` |
| 진행 상황을 음성으로 안내·해설 | 멀티에이전트 + 음성 | `agents/`, `audio/` |
| 현재 상태를 화면에 표시 | 태블릿 UI | `frontend/` |

각 구성 요소를 연결·조립하는 계층은 서버(`backend/`)와 비전–규칙 통신 계층(`bridge/`)이다.

### 1.1 핵심 개념

구현에 앞서 반드시 이해해야 하는 세 가지 개념을 정리한다.

**FSM (유한 상태 기계, Finite State Machine)**
게임은 일련의 단계(상태)로 구성된다. 예를 들어 요트다이스의 한 턴은
`굴림 대기 → 킵 선택 → 점수 기록 → 다음 플레이어` 순으로 진행된다.
FSM은 현재 어느 단계인지를 보관하는 객체이며, 이벤트(예: 주사위 굴림)가 입력되면
규칙에 따라 상태를 전이시키고, 화면 갱신 메시지를 생성하여 반환한다.

**GameEvent 와 FusionContext (비전과 규칙이 교환하는 두 가지 메시지)**
비전과 FSM은 서로의 함수를 직접 호출하지 않고, 두 종류의 데이터만 교환한다.

- `GameEvent` — 비전 → FSM 방향. "특정 사건이 발생했음"을 통지한다.
  예: `ROLL_CONFIRMED`(주사위 굴림 확정), `werewolf_vote_point`(특정 대상 지목).
- `FusionContext` — FSM → 비전 방향. "현재 단계에서 기대하는 입력"을 통지한다.
  예: 현재 차례 플레이어, 허용된 행위자, 기대 이벤트 목록.

양방향 구조를 채택한 이유는 오인식을 억제하기 위함이다. 비전이 관찰한 모든 사건을 보고하면
오인식이 누적되므로, FSM이 각 상태에서 기대하는 이벤트를 사전에 한정하고
비전은 그 외의 사건을 발화하지 않는다. 이 **상태 기반 게이팅**이 시스템 정확도의 근간이며,
새 게임을 추가할 때에도 이 계약만 준수하면 된다.

**파이프라인(Pipeline)과 스레드**
카메라는 한 대이나, 그 영상을 여러 인식기가 동시에 처리한다.
`로비(손 인식)`, `요트(주사위 인식)`, `늑대인간(카드 인식)` 파이프라인이 각각 별도 스레드에서 동작하며,
현재 진행 중인 게임의 파이프라인 하나만 실제 연산을 수행한다(나머지는 대기).
따라서 새 게임을 추가하는 작업은 새 파이프라인을 하나 추가하는 작업을 포함한다.

## 2. 처리 흐름 예시 (요트 주사위 굴림)

전체 동작을 이해하기 위해, 기존 게임에서 하나의 사건이 화면에 반영되기까지의 경로를 정리한다.
플레이어가 주사위를 굴리는 시점부터의 흐름은 다음과 같다.

```
1. 카메라가 프레임을 캡처 → 요트 파이프라인(vision/yacht/pipeline.py)이 수신.
2. YOLO가 주사위·트레이·굴림통을 검출, ByteTrack이 각 주사위에 추적 번호를 부여,
   DotCounter가 주사위 눈(1~6)을 판독, MediaPipe가 손을 검출.
3. RollAttributor가 "손이 트레이에 진입했다가 이탈했고 주사위 눈 분포가 변경됨 → 굴림 확정"으로 판정.
4. FusionEngine이 "현재 FSM이 ROLL_CONFIRMED를 기대하는지" 확인 후, 조건 충족 시 GameEvent 발화.
5. GameEvent가 Bridge → YachtRunner → 현재 게임 Session으로 전달.
6. 심판 에이전트가 굴린 사람이 현재 차례인지 우선 검사.
7. YachtFSM이 ROLL_CONFIRMED를 처리하여 상태를 '킵 선택'으로 전이하고,
   새 FusionContext와 화면 갱신 메시지를 생성.
8. Session이 메시지를 분배: FusionContext는 비전으로(기대 이벤트 갱신), 화면·음성은 태블릿으로.
9. 진행 에이전트가 "기회가 두 번 남았습니다" 등의 멘트를 합성하여 태블릿에서 재생.
10. 태블릿 화면이 새 주사위 눈으로 갱신.
```

새 게임을 추가하는 작업은 위 흐름에서 `yacht`로 표기된 구성 요소를 새 게임의 것으로 대체하는 것과 같다.

## 3. 구현 — 여섯 개 구성 요소

하나의 게임은 다음 여섯 개 레이어에 걸쳐 구현된다. 순서는 2절의 처리 흐름과 동일하다.

```
 ① 게임 규칙   games/<game>/fsm.py + state 등              이벤트 → 상태 전이 → 화면/음성 메시지
 ② 비전 인식   vision/<game>/pipeline.py                  프레임 → 인식 결과
 ③ 융합 규칙   vision/fusion/<game>_rules.py               인식 결과 → GameEvent 후보
 ④ 배선        backend/<game>_runner/session + server.py   ①②③을 서버에 연결
 ⑤ 진행/음성   agents/progress_agent.py + audio/catalog.py 진행 멘트
 ⑥ 화면        frontend/src/pages/<Game>.jsx 등            태블릿 UI
 (+ 자산)      weights/<game>_vN.pt                        YOLO 가중치
```

> 권장 사항: 처음부터 새로 작성하기보다, 성격이 가장 유사한 기존 게임을 복사하여 명칭과 내용을 수정한다.
> - 물체를 YOLO로 인식하는 게임(주사위·말 등) → `yacht` 복사.
> - 카드를 인식하는 게임 → `werewolf` 복사.
> - 손동작/제스처만 사용하는 단순한 게임 → `vision/lobby/pipeline.py`가 최소 골격을 제공한다.

각 레이어를 "목적 → 구현 대상 → 복사 출발점" 순으로 설명한다.

### 3.1 게임 규칙 — `games/<game>/`

목적: 게임의 단계 흐름과 처리해야 할 이벤트를 정의한다. 규칙을 먼저 확정해야 비전이 무엇을
인식해야 하는지도 결정되므로, 이 레이어를 가장 먼저 구현한다.

구현 대상(요트 또는 늑대 폴더 복사):

- `ontology.py`(또는 요트와 같이 `state.py` 내부) — 게임 전용 단계·이벤트·입력 종류를 `StrEnum` 상수로 정의.
  ```python
  class MyGamePhase(StrEnum):
      AWAITING_MOVE = "awaiting_move"
      GAME_END = "game_end"

  class MyGameEventType(StrEnum):
      PIECE_MOVED = "mygame_piece_moved"   # 비전이 발화할 이벤트 명
  ```
- `state.py` — 게임 상태를 보관하는 데이터 객체. 점수·차례·보드 상태 등과 함께
  `to_dict()`(화면 전송용)와 `new(players)`(초기화) 메서드를 제공한다.
- `fsm.py` — 게임 규칙. `games/base_fsm.py`의 `BaseFSM`을 상속하여 다음 네 메서드를 구현한다.
  ```python
  class MyGameFSM(BaseFSM):
      def handle_event(self, event): ...        # 비전 이벤트 → 상태 전이 + 메시지 반환
      def handle_input(self, input_type, data, player_id=None): ...  # UI 입력 처리
      def get_fusion_context(self): ...         # 현재 단계의 기대 컨텍스트 반환
      def get_state_dict(self): ...             # 화면에 전송할 현재 상태
  ```
- (필요 시) 순수 계산 함수 — 요트 `scoring.py`(점수 계산), 늑대 `judge.py`(승패 판정) 등.

`get_fusion_context()`는 비전에 전달할 컨텍스트를 생성하는 핵심 메서드이다.
각 단계에서 현재 차례 플레이어와 기대 이벤트를 지정한다.
```python
def get_fusion_context(self):
    return FusionContext(
        fsm_state=self.state.phase,                     # 현재 단계
        game_type="mygame",                             # 게임 명 (비전이 이 값으로 구독)
        active_player=self.state.current_player.player_id,
        allowed_actors=[현재_차례_player_id],            # 허용 행위자 → 차례 위반 자동 감지
        expected_events=[MyGameEventType.PIECE_MOVED],  # 이 단계에서 수신할 이벤트
        reject_events=[...],                            # 명시적으로 거부할 이벤트
    )
```

규칙: 모든 상태 전이 시 `self.state.state_version += 1`을 수행한다.
각 메서드는 일반적으로 `[화면 갱신 메시지, fusion_context 메시지]`를 반환한다.

검증: `tests/test_<game>_fsm.py`에 단계 전이 및 이벤트 처리에 대한 단위 테스트를 추가한다.

### 3.2 비전 인식 — `vision/<game>/`, `vision/fusion/<game>_rules.py`

목적: 카메라가 캡처한 영상을 ①에서 정의한 이벤트로 변환한다.

검출·추적 구성 요소는 이미 구현되어 있으므로 새로 작성하지 않고 조립하여 사용한다.

- 물체 검출: `YoloDetector`(학습된 `.pt` 모델 필요)
- 손 검출: `HandDetector`(MediaPipe), `GestureClassifier`(주먹/브이/OK 등)
- 추적: `ByteTracker`(물체 번호 유지), `CardTracker`/`DiceManager`
- 행위자 식별: `seat_matcher`, `geometry/arm_vector`(팔 방향 기반 플레이어 식별)

구현 대상(요트/늑대 `vision/<game>/` 복사):

- `config.py` — 가중치 경로, YOLO 설정, 프레임 스킵 등 설정값 정의.
- `pipeline.py` — 위 구성 요소를 조립하는 메인 루프.
  - `__init__`: 필요한 검출기·추적기 생성 후
    `self._bridge.on_fusion_context(self._on_fusion_context, game_type="mygame")`로
    해당 게임의 컨텍스트만 구독한다.
  - `start(frame_queue)`: 카메라 큐에서 프레임을 받아 매 프레임 처리하는 블로킹 루프.
  - `set_active(enabled)`: 해당 게임 진행 중에만 True(그 외에는 프레임만 소진).
- `vision/fusion/<game>_rules.py` — 인식 결과를 이벤트 후보로 변환하는 규칙(핵심 파일).
  ```python
  class MyGameRules:
      def build_candidates(self, ctx, perception):
          # perception(현재 프레임 인식 결과)을 분석하여 이벤트 후보 목록 반환.
          # 반환 형식: [(event_type, data_dict, confidence), ...]
          # data_dict에는 actor_id(행위자)와 안정화 비교용 "_key"를 포함한다.
          return [("mygame_piece_moved", {"actor_id": pid, "_key": (...)}, 0.9)]
  ```
  중복 발화를 방지하기 위해 발화 이력 집합(`self._reported_*`)을 유지하고, 단계 전환 시 초기화한다.

FusionEngine 연결: 파이프라인 `__init__`에서 `self._fusion.register_<game>_rules(rules)`로 규칙을 주입한다
(늑대인간의 `register_werewolf_rules` 방식을 참고).

FusionEngine이 자동으로 처리하는 검증: 후보가 이벤트로 확정되려면 세 조건을 통과해야 하며,
이는 엔진이 수행한다. ① FSM이 기대하는 이벤트인지 ② 실제 변화가 있었는지
③ N프레임 동안 동일 판정이 유지되었는지(떨림 방지). 따라서 `build_candidates`는
현재 프레임에서 관찰된 사건만 보고하면 된다.

가중치: 새 물체를 인식해야 하는 경우 YOLO를 학습하여 `weights/<game>_vN_YYYYMMDD.pt`로 배치한다.
가중치 파일은 git에 커밋하지 않으며 Google Drive로 공유한다(`weights/README.md` 참고).
모델 부재 시 해당 인식은 빈 결과로 동작한다.

### 3.3 서버 배선 — `backend/`

목적: ①②③을 구현하더라도, 서버가 "해당 게임 시작 시 파이프라인을 활성화하고 이벤트를 세션으로 전달"하도록
연결하지 않으면 동작하지 않는다. 이 연결은 수동으로 수행한다(자동 등록 메커니즘은 존재하지 않는다).
이 단계의 누락이 빈번하므로 주의한다.

구현 대상(요트 모듈 복사):

- `backend/<game>_runner.py` — 비전 파이프라인을 백그라운드 스레드로 실행하고, 비전이 발화한 이벤트를
  현재 게임 세션으로 전달하는 라우터.
- `backend/<game>_session.py` — `/ws/<game>` WebSocket 연결을 담당. UI 입력을 FSM에 전달하고,
  FSM이 생성한 메시지를 태블릿·비전·오디오로 분배한다. 에이전트와 오디오 연결도 이 모듈에서 수행한다.

`backend/server.py`의 `lifespan` 함수에 다음 다섯 위치를 추가한다(모두 필요).
```python
# 1) 러너 생성
mygame_runner = MyGameRunner(config=..., bridge=bridge, loop=loop)

# 2) 플레이어 목록 변경 통지
def _on_players_changed(players):
    ...
    mygame_runner.update_players(players)

# 3) 게임 전환 시 파이프라인 활성/비활성
def _on_game_switch(game_type):
    ...
    mygame_runner.set_active(game_type == "mygame")

# 4) 카메라 큐 구독 및 스레드 시작
mygame_queue = camera.subscribe()
mygame_runner.start(mygame_queue)
app.state.mygame_runner = mygame_runner

# 5) WebSocket 엔드포인트 추가 (요트/늑대 엔드포인트 참고)
@app.websocket("/ws/mygame")
async def mygame_socket(websocket):
    # MyGameSession 생성 → register_session → 메시지 루프 → finally 정리
    ...
```

### 3.4 진행 및 음성 — `agents/`, `audio/`

목적: FSM은 상태만 전이하며 음성을 발화하지 않는다. 음성 안내는 진행 에이전트가 전담한다.

- `agents/progress_agent.py` — 각 단계에서 발화할 멘트를 정의한다.
  - 늑대인간과 같이 고정 대사를 사용하는 경우, `_MYGAME_SCRIPTS = {"단계명": "발화 내용"}` 사전을
    추가하고 `_SCRIPTS`에 등록한다.
  - 요트와 같이 상황별로 다른 멘트가 필요한 경우, FSM의 `state.last_message`를 그대로 발화하도록 분기한다.
- `audio/catalog.py` — `STATIC_LINES`에 고정 멘트(플레이어 이름이 포함되지 않는 문장)를 등록하면
  서버 기동 시 사전 합성되어 지연 없이 재생된다. 에이전트 대사와 문자열이 정확히 일치해야 캐시가 적중한다.
- (선택) 효과음·배경음은 `SFX_REGISTRY`/`BGM_REGISTRY`에 등록한다.
- 심판(`rules_agent`)·템포(`tempo_agent`) 에이전트는 일반적으로 수정이 불필요하다.
  FSM이 `allowed_actors`(차례 정보)와 타이머만 제공하면 차례 안내 및 시간 경고가 자동으로 동작한다.

### 3.5 화면 — `frontend/`

- `frontend/src/pages/MyGameGame.jsx` — `useWebSocket("/ws/mygame")`로 서버에 연결한다.
  서버가 전송하는 `state_update`로 화면을 렌더링하고, 사용자 입력은 `send(input_type, data, player_id)`로 전송한다.
- `frontend/src/components/mygame/` — 단계별 화면 컴포넌트.
- 로비 화면(`pages/Lobby.jsx`)의 게임 선택 목록에 새 게임을 추가한다.
- 음성은 `useAudioPlayer`가 자동 재생하며 종료 시 서버로 `audio_ack`를 전송한다(별도 구현 불필요).

## 4. 권장 작업 순서

전체를 한 번에 구현하기보다, 단계별로 검증하며 확장하는 방식을 권장한다.

1. **규칙 우선 구현** — `games/<game>/`의 FSM을 작성하고 `tests/test_<game>_fsm.py`로 검증한다.
   비전 없이 테스트에서 이벤트를 직접 주입하여 단계 전이를 확인한다.
2. **서버 연결 및 화면** — `backend/`와 `frontend/`를 연결하고, 화면 버튼으로 이벤트를 모의 입력하여
   게임이 끝까지 진행되는지 확인한다. 요트 세션의 `MANUAL_DICE_INPUT`과 같이 비전을 대체하는
   수동 입력 경로를 먼저 마련하면 비전 없이 전체 흐름을 검증할 수 있다.
3. **비전 연결** — 마지막으로 `vision/<game>/`, `_rules.py`, YOLO 가중치를 추가하여 실제 카메라로 동작시킨다.

이 순서를 따르면 각 단계를 독립적으로 검증할 수 있어 문제 구간을 좁히기 쉽다.

## 부록 A. 기존 게임의 구성 요소 대응표

새 게임을 구현할 때 동일한 역할을 하는 기존 파일의 위치를 참조한다.

| 레이어 | 요트다이스(`yacht`) | 한밤의 늑대인간(`werewolf`) |
|---|---|---|
| 규칙(FSM) | `games/yacht/{fsm,state,scoring}.py` | `games/werewolf/{fsm,state,ontology,judge,night_roles}.py` |
| 비전 파이프라인 | `vision/yacht/{pipeline,config,schemas}.py` | `vision/werewolf/{pipeline,config,schemas}.py` |
| 융합 규칙 | `vision/fusion/yacht_rules.py` (+ `attribution/roll_attributor.py`) | `vision/fusion/werewolf_rules.py` (+ `tracking/card_tracker.py`) |
| 배선(러너/세션) | `backend/{yacht_runner,yacht_session}.py` | `backend/{werewolf_runner,werewolf_session}.py` |
| 진행 멘트 | ProgressAgent 동적(`last_message`) | ProgressAgent `_WEREWOLF_SCRIPTS` |
| 화면 | `frontend/src/pages/YachtGame.jsx` | `frontend/src/pages/WerewolfGame.jsx` + `components/werewolf/` |
